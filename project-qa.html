Q1 整体数据流是怎样的？ ▼

端到端流水线如下：

  1. 帧采集 — cv::VideoCapture 从摄像头/视频/图片读取
  2. 人脸检测 — dlib HOG 检测器找到所有人脸矩形
  3. 选取最大人脸 — 面积最大的人脸作为目标
  4. 提取 68 关键点 — dlib shape_predictor 输出轮廓/眉毛/眼睛/鼻子/嘴巴坐标
  5. 裁剪人脸区域 — 用关键点边界框裁剪并 resize 到 224×224
  6. ONNX 推理 — HSEmotion EfficientNet-B0 输出 7 类 softmax 概率
  7. 5 帧平滑 — 滑动窗口多数投票，过滤抖动
  8. 情绪 → 音乐参数 — EmotionMapper 映射调性/音阶/速度/音色
  9. 生成旋律+伴奏 — MusicGenerator 根据参数合成完整曲目
  10. WAV 写盘 + 播放 — AudioPlayer 生成 WAV，fork() 子进程调 paplay

同时，主线程每帧把帧数据与情绪数据写入 SharedState（mutex 保护），Web 线程读取后推送 MJPEG 流并提供 REST API。

Q2 为什么选 dlib HOG 而不是深度学习人脸检测？ ▼
  • 无需 GPU — HOG 是传统特征方法，纯 CPU 推理快（~10-30ms）
  • 依赖少 — dlib 一个库搞定检测+关键点，部署简单
  • 正脸精度足够 — 课程项目场景下正脸为主，HOG 够用
  • 对比 — OpenCV DNN / RetinaFace / MTCNN 虽更准，但引入额外模型文件和推理框架，增加复杂度
Q3 HSEmotion 模型是什么？为什么用它？ ▼

HSEmotion 是基于 EfficientNet-B0 微调的人脸情绪识别模型。

  • 轻量 — 参数量仅 ~5M，CPU 推理可达实时
  • 7 类输出 — Angry / Disgust / Fear / Happy / Sad / Surprise / Neutral，覆盖基本情绪
  • ONNX 格式 — 导出为 ONNX，用 onnxruntime 推理，不依赖 PyTorch/Python 运行时
  • 高精度 — 在 AffectNet / RAF-DB 数据集上准确率高
Q4 情绪是怎么映射到音乐参数的？ ▼

通过 EmotionMapper 模块，每种情绪映射到不同参数组合：

情绪调性/音阶速度 BPM音色特征
Happy 开心C 大调快速明亮钢琴（多泛音）
Sad 悲伤D 小调慢速柔和弦乐（少泛音）
Angry 愤怒减音阶急促刺耳（失真泛音）
Fear 恐惧—不规律拨弦（强攻击）
Surprise 惊讶—快速明亮钢琴
Disgust 厌恶—中速暗色垫音（阻尼）
Neutral 中性—中等温和钢琴（平衡）

Q5 项目有几个线程？各自职责是什么？ ▼
执行单元类型职责
主线程std::thread (主)采集帧 → 人脸检测 → 情绪识别 → UI 渲染 → 音乐生成控制，循环执行
Web 线程std::thread运行 POSIX socket HTTP 服务器，处理 MJPEG 推流和 REST API

严格来说是 2 个线程 + 1 个子进程。

Q6 线程间怎么通信？SharedState 的设计？ ▼

SharedState 使用两种同步机制：

  • mutex（互斥锁） — 保护复杂数据：latest_frame（cv::Mat 帧）、情绪概率 confidences、音乐参数等。主线程写，Web 线程读，用 std::lock_guard RAII 自动加解锁
  • atomic（原子变量） — 控制标志：play_requested、pause_requested、auto_toggle_requested、mode_switch_requested 等，无锁操作

设计模式是典型的 生产者-消费者：主线程生产帧和情绪数据，Web 线程消费。atomic 标志实现反向控制（Web → 主线程）。

Q7 为什么不全用 mutex，而要区分 mutex 和 atomic？ ▼
  • mutex 适合保护复杂数据结构（cv::Mat、vector），需要临界区保证一致性
  • atomic 适合简单的布尔标志，无锁、开销极小
  • Web 线程 store(true) 置位，主循环 load() 检查后 store(false) 清除，整个过程不需要加锁
  • atomic 的内存序（memory order）保证了跨线程的可见性

简言之：数据用锁，标志用原子，各取所长。

Q8 MJPEG 推流为什么限制只能一个客户端？ ▼

用 mjpeg_busy_ atomic 标志控制。原因：

  • 每个客户端都要 JPEG 编码 + 发送帧数据，多客户端倍增 CPU 和带宽开销
  • JPEG 编码是 CPU 密集操作，多个客户端会拖慢主循环帧率
  • 实际场景中手机/浏览器远程查看，单客户端已满足需求

Q9 实时性怎么保证？FPS 多少？ ▼

主循环目标 30 FPS（每帧 <33ms）。典型耗时分布：

步骤耗时
dlib HOG 人脸检测~10-30ms
ONNX 情绪推理~10-20ms
UI 渲染（OpenCV 绘制）~2-5ms
JPEG 编码（MJPEG）~3-5ms

代码中有 FPS 计数器（每 1000ms 重置），实际 i5 及以上 CPU 可达 15-30 FPS。

Q10 如果人脸检测太慢怎么办？ ▼
  • 跳帧检测 — 不是每帧都跑检测，可以隔 N 帧检测一次，中间帧沿用上次结果
  • 调小 pyramid 参数 — HOG 的金字塔缩放步长调大，牺牲精度换速度
  • 只取最大人脸 — 避免多人脸的额外处理开销
  • 降低输入分辨率 — 缩小帧尺寸再检测
Q11 音乐生成会阻塞主线程吗？ ▼

不会完全阻塞。策略如下：

  • 生成完整 WAV 文件（旋律+伴奏+混响）确实耗时，但频率低（每 5 秒自动播放一次）
  • 音频播放通过 fork()+exec 创建子进程，父进程不等待子进程结束（非阻塞）
  • 生成动作在主循环中执行，但由于频率低，对整体帧率影响有限
Q12 ONNX Runtime 为什么比 PyTorch 推理快？ ▼
  • 计算图优化 — 算子融合、常量折叠、死代码消除
  • 静态图 — ONNX 导出后是固定计算图，推理路径确定，可做更多编译期优化
  • 无 Python 开销 — 去掉了 Python 解释器和动态图构建开销
  • 优化的算子实现 — CPU 提供者使用 SIMD/多线程优化的算子
  • 内存规划 — 可预分配内存，减少运行时动态分配

四、音频合成原理
2 题
▼
Q13 钢琴音色是怎么合成的？ ▼

不是采样回放，而是物理建模合成，5 个关键步骤：

  1. 三弦模型 — 每个音模拟 3 根略微走音的弦（频率比 0.9994 / 1.0 / 1.0006），产生自然的拍频效果（beating）
  2. 双段 ADSR 包络：
    • 快速初始衰减 — 衰减率 8.0，约 80ms
    • 慢速尾部衰减 — 衰减率 0.8，时长与音高负相关（低音更长）
  3. 泛音叠加 — 根据力度添加 6-12 个泛音，力度越大泛音越多（音色越亮）
  4. 击弦瞬态噪声 — 模拟琴锤敲击弦时的短暂噪声爆发
  5. Schroeder 混响 — 4 个梳状滤波器 + 2 个全通滤波器模拟空间混响
Q14 Schroeder 混响算法的原理？ ▼

经典人工混响算法，分两级处理：

  1. 梳状滤波器（Comb Filter）× 4
    • 不同延迟长度的反馈延迟线
    • 产生密集的回声，模拟房间早期反射
    • 延迟时间不同避免"金属感"
  2. 全通滤波器（Allpass Filter）× 2
    • 打乱梳状滤波器产生的频率"染色"
    • 让混响更平滑、自然、扩散

可调参数：混响时间（RT60）、干湿比（dry/wet mix）。

Q15 为什么不用第三方库而自己写 HTTP 服务器？ ▼
  • 课程要求 — 展示对底层网络编程的理解（POSIX socket API）
  • 需求简单 — 只需要 MJPEG 推流 + 几个 REST 端点，不需要重量级框架
  • 依赖最小化 — 直接用 socket() / bind() / listen() / accept()，无额外安装
  • 代码量可控 — 全部在 web_server.h/.cpp 两个文件中
Q16 MJPEG 推流的技术原理？ ▼

MJPEG (Motion JPEG) 基于 HTTP 长连接实现：

  1. 服务器返回响应头：Content-Type: multipart/x-mixed-replace; boundary=frame
  2. 每帧发送：--frame\r\nContent-Type: image/jpeg\r\nContent-Length: N\r\n\r\n + JPEG 二进制数据
  3. 浏览器收到新帧自动替换旧帧，形成"视频流"
  4. 服务器用 select() 非阻塞轮询，66ms 间隔推帧（~15 FPS）
  5. JPEG 压缩质量设为 65，平衡画质和带宽

Q17 5 帧滑动窗口是怎么做情绪平滑的？ ▼

保存最近 SMOOTH_WINDOW = 5 帧的识别结果，取众数（majority voting）作为最终情绪。

例如：连续 4 帧识别为 Happy，1 帧误判为 Neutral，投票后仍输出 Happy。

优点：过滤偶发误识别，让情绪切换更稳定；缺点：引入约 5 帧延迟（~170ms @30FPS），可忽略不计。

Q18 如果没有人脸怎么办？ ▼

主循环检查 faces.empty()：

  • 无人脸时跳过情绪识别和音乐生成，只渲染 UI
  • SharedState 中 face_detected 置 false，Web 端能看到"未检测到人脸"
  • 保持上一次有效情绪数据，不重置为中性
Q19 怎么处理光照和角度变化？ ▼
  • HOG 特征 对正脸效果好，大角度侧脸可能检测不到（固有限制）
  • HSEmotion 训练数据 有一定光照/角度增强，模型本身有一定鲁棒性
  • 极端条件下准确率会下降，这是 该方案的固有局限
  • 改进方向：可换成 CNN 人脸检测器（如 RetinaFace）+ 更大数据训练的模型
Q20 fork 播放音频会不会产生僵尸进程？怎么解决？ ▼

会的。fork() 后如果不调用 wait()/waitpid()，子进程结束后变成僵尸进程（Z 状态）。

解决方案：

  • signal(SIGCHLD, SIG_IGN) — 让内核自动回收子进程，最简单
  • 或者主循环中非阻塞调用 waitpid(-1, nullptr, WNOHANG) 定期清理
  • 或者用 prctl(PR_SET_PDEATHSIG, SIGKILL) 在子进程中设置父进程退出时自己也被杀

Q22 关键路径（性能瓶颈）在哪里？ ▼

主循环关键路径（按耗时从高到低）：

瓶颈 #1 dlib HOG 人脸检测 — ~10-30ms，占整帧 40-60% 最大瓶颈
瓶颈 #2 ONNX 情绪推理 — ~10-20ms，占整帧 30-40%
开销 #3 JPEG 编码（MJPEG 推流时）— ~3-5ms
开销 #4 UI 渲染（OpenCV 绘制）— ~2-5ms

人脸检测 + 情绪推理合计占 70-90% 的帧处理时间，是绝对的关键路径。其余步骤（UI 绘制、状态更新、按键处理）开销很小。

Q23 具体用了哪些优化方法？各自原理？ ▼

算法层优化：

  • 跳帧检测 — 不每帧都跑 HOG 检测，隔 N 帧检测一次，中间帧复用上次人脸位置。原理：人脸位置帧间变化很小，无需逐帧重检测
  • 只取最大人脸 — 检测到多张人脸后只对面积最大的做情绪识别。原理：画面中最大人脸通常是主要用户
  • 5 帧多数投票 — 滑动窗口取众数。原理：利用时序连续性过滤单帧噪声

系统层优化：

  • atomic 替代 mutex — 简单标志用 std::atomic<bool>，无锁。原理：atomic 利用 CPU CAS 指令，避免互斥锁的内核态切换开销
  • fork() 非阻塞播放 — 子进程执行 paplay，父进程继续主循环。原理：进程级并行，音频 I/O 不阻塞视觉处理
  • JPEG 质量 65 — 压缩质量设为 65（0-100）。原理：65 是视觉质量与文件大小的甜点，再高人眼几乎看不出差别但编码耗时增加
  • 单客户端 MJPEG — mjpeg_busy_ atomic 互斥。原理：避免重复编码同帧，O(1) 客户端开销
  • ONNX 静态图 — 用 onnxruntime 替代 PyTorch 动态图。原理：固定计算图可做算子融合、内存预分配等编译期优化

Q25 图像处理各环节是有损还是无损的？ ▼
处理环节有损/无损说明
摄像头采集（BGR 矩阵）有损传感器本身有噪点、拜耳滤镜插值，非真实场景完美复现
dlib HOG 特征提取有损HOG 将图像转换为梯度方向直方图，丢弃了纹理/颜色等细节，只保留形状信息
68 关键点提取有损整张人脸压缩为 68 个 (x,y) 坐标，丢弃了绝大多数像素信息
人脸裁剪 + resize 到 224×224有损降采样丢失高频细节，双线性/双三次插值会模糊边缘
归一化（/255 减均值除标准差）无损线性变换，数学上可逆（知道均值/标准差可还原），信息量不变
ONNX softmax 推理无损确定性浮点计算，相同输入永远产生相同输出，无随机丢失
5 帧投票平滑有损丢弃了少数帧的识别结果（被投票覆盖），但这是设计意图
JPEG 编码（MJPEG 推流）有损JPEG 是有损压缩格式，DCT 变换 + 量化丢弃高频分量，quality=65 丢弃约 35% 信息
帧克隆（clone 给 Web 线程）无损cv::Mat::clone() 是像素级深拷贝，数据完全一致

关键结论：检测和识别环节都是有损的（这是特征提取的本质 — 从高维数据中提取低维语义），但推理过程本身是确定性的无损计算。JPEG 推流是唯一"刻意损失画质"的环节，为了带宽做的取舍。

Q26 项目中有没有用性能探针（Profiling）？怎么做性能分析？ ▼

项目内置探针：

  • FPS 计数器 — 主循环中用 std::chrono::steady_clock 计时，每 1000ms 重置一次，实时显示在 UI 状态栏
  • 分段计时探针（已实现） — 8 个管线阶段各插入 ScopedTimer RAII 守卫，自动记录 last/avg/max 三项指标
  • 性能面板 — 按 P 键弹出右上角毛玻璃面板，实时显示各阶段耗时 + 颜色编码进度条 + 帧总耗时
  • Web API 暴露 — /api/status JSON 中新增 perf 字段，远程可查看各阶段耗时

探针架构（PerfProfiler）：

  • perf_profiler.h — Header-only 单例，零分配热路径
  • ScopedTimer RAII 守卫 — 放在代码块开头，出作用域自动记录耗时
  • StageTiming — 每阶段存储 last_ms / avg_ms（指数滑动平均 alpha=0.1）/ max_ms
  • 线程安全：仅主线程写入，Web/UI 线程读取（对陈旧数据容忍）
